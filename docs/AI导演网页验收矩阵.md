# AI 导演网页验收矩阵

更新时间：2026-04-28

## 1. 目的

这个矩阵用于约束每个网页任务包的验收方式。以后不再用“看起来还行”作为完成标准，而是按视口、状态、交互、可访问性和性能风险逐项检查。

## 2. 开源方法转化

### Playwright 思路：真实浏览器 + 多视口

参考：[microsoft/playwright](https://github.com/microsoft/playwright)

转化为本项目规则：

- 每个 UI 任务至少检查桌面和手机两个视口。
- 每个核心流程至少检查一次真实点击路径。
- 截图不是为了像素完美，而是为了发现明显布局错误、溢出、重叠和状态缺失。

当前阶段不安装 Playwright，先用 Codex in-app browser 做人工截图 QA。后续如用户确认，再安装并自动化。

### Storybook 思路：状态隔离

参考：[storybookjs/storybook](https://github.com/storybookjs/storybook) 与 [Storybook Visual Testing Handbook](https://storybook.js.org/tutorials/ui-testing-handbook/react/en/visual-testing)

转化为本项目规则：

- 每个模块都必须列出状态，而不是只做有数据状态。
- 每个模块至少覆盖：空、生成中、已生成、失败/超时。
- 对当前原生页面，先用“状态样例清单”替代 Storybook。

### axe-core 思路：自动可访问性规则

参考：[dequelabs/axe-core](https://github.com/dequelabs/axe-core)

转化为本项目规则：

- 输入框必须有 label。
- 按钮文案必须能单独说明动作。
- 状态不能只靠颜色表达。
- 图片/视频必须有可理解的替代信息或上下文说明。
- 标题层级不能混乱到影响扫描。

当前阶段不安装 axe-core，先人工检查。后续如用户确认，再接入自动检查。

### Lighthouse 思路：性能和可访问性审计

参考：[GoogleChrome/lighthouse](https://github.com/GoogleChrome/lighthouse)

转化为本项目规则：

- 大量 LibTV 素材不能无限自动加载。
- 图片和视频要限制展示数量，超出部分折叠。
- 首页不做大营销图，不增加无关重资源。
- 内测工具优先性能和可用性，不优先 SEO。

## 3. 固定视口

每个 UI 任务至少检查：

| 视口 | 用途 | 验收重点 |
| --- | --- | --- |
| 1440 x 900 | 桌面工作台 | 左侧输入 + 右侧结果能同屏理解；工作台信息密度合理 |
| 390 x 844 | 手机 | 单列布局；按钮不挤压；结果总览不被最近任务压到太下面 |

后续可选补充：

| 视口 | 用途 | 验收重点 |
| --- | --- | --- |
| 1024 x 768 | 小屏桌面/平板 | 左右布局是否过早拥挤 |
| 1920 x 1080 | 大屏 | 内容不要散得太开，剪辑面板仍像工具 |

## 4. 固定状态

每个任务包需要尽量覆盖以下状态：

| 状态 | 用户看到什么 | 常见风险 |
| --- | --- | --- |
| 无任务 | 输入区、空结果提示 | 用户不知道输入后会得到什么 |
| 生成中 | 明确 loading 文案、进度阶段 | 用户以为卡住 |
| 有剧本无素材 | 剧本、角色/场景/道具/分镜草稿 | 用户误以为素材应该已经生成 |
| 有素材未成片 | 图片/视频素材、选用状态、剪辑面板 | 素材太多撑坏页面 |
| 已成片 | 视频播放器、画布链接、可继续调整 | 最终视频和步骤素材关系不清 |
| 失败/超时 | 可理解错误、同步入口 | 错误太技术化或像主流程彻底坏掉 |
| 长文本 | 长剧本、长 prompt、长文件名 | 卡片撑破、按钮被挤出 |

## 5. 任务包验收表

### 任务 0：Agent 执行规范

| 检查项 | 通过标准 |
| --- | --- |
| 开发前阅读 | 明确列出页面规格、推进计划、当前任务包 |
| 禁止事项 | 明确不自动安装第三方插件、不大范围重构 |
| 完成汇报 | 有改动、验证、剩余问题、下一步 |
| 截图 QA | 明确桌面和手机至少各检查一次 |

### 任务 1：首屏和结果总览

| 检查项 | 通过标准 |
| --- | --- |
| 第一屏信息 | 桌面端能同时看到输入区和当前任务核心状态 |
| 结果总览 | 有剧本、角色、场景、道具、分镜、成片六块 |
| 查看入口 | 每块能跳到对应模块 |
| 空状态 | 没有任务时说明输入后会生成什么 |
| 移动端 | 输入区后紧跟结果总览，最近任务不抢第一屏 |
| Provider 状态 | GPT-5.4 和 LibTV ready 清楚；备用缺失不误导 |

### 任务 2：剧本可视化

| 检查项 | 通过标准 |
| --- | --- |
| 标题和梗概 | GPT 返回后立刻显示 |
| 旁白 | 整片旁白可读，不挤压卡片 |
| 文字分镜 | 至少 3 条，含编号、标题、画面说明、旁白 |
| 调整入口 | 用户能自然语言改剧本，不必触发 LibTV |
| 长文本 | 长分镜说明不溢出 |

### 任务 3：角色/场景/道具主体图

| 检查项 | 通过标准 |
| --- | --- |
| 草稿卡 | 每组至少一张卡，名称、设定、LibTV 任务清楚 |
| 保存 | 修改文字后能保存 |
| 单卡生成 | 单张卡能交给 LibTV |
| 本组生成 | 本组能批量生成 |
| 素材展示 | 返回图片或视频后显示在卡片内 |
| 选用 | 多个素材能选用一个，选用状态明显 |

### 任务 4：分镜到视频镜头

| 检查项 | 通过标准 |
| --- | --- |
| 分镜字段 | 每条有时长、镜头、旁白、LibTV 任务 |
| 单镜头生成 | 每条分镜能单独生成 |
| 按顺序生成 | 有明确按钮和文案说明 |
| 视频预览 | 返回视频后可播放 |
| 时间线 | V1 按分镜顺序显示状态 |

### 任务 5：剪辑面板

| 检查项 | 通过标准 |
| --- | --- |
| 素材池 | 展示已返回素材，选用素材排前 |
| 预览窗口 | 有最终视频时播放最终视频，无素材时有占位 |
| 检查器 | 片长、素材数、分镜数清楚 |
| 时间线 | V1/A1 稳定，不随文字撑坏 |
| 移动端 | 单列可用，时间线可横向滚动 |

### 任务 6：最终成片和同步

| 检查项 | 通过标准 |
| --- | --- |
| 提交成片 | 点击后有等待状态 |
| 画布链接 | 有 projectUrl 时显示 |
| 手动同步 | 超时或等待时可点击同步 |
| 视频播放 | 返回视频后播放器可用 |
| 错误文案 | 用户能理解，不暴露 key 或堆栈 |

### 任务 7：截图 QA 和视觉修复

| 检查项 | 通过标准 |
| --- | --- |
| 桌面截图 | 1440 x 900 无明显重叠和溢出 |
| 手机截图 | 390 x 844 无横向滚动 |
| 空状态 | 用户知道下一步 |
| 生成中状态 | 用户知道正在做什么 |
| 已生成状态 | 用户知道结果在哪 |
| 长文本 | 不撑破按钮、卡片、时间线 |
