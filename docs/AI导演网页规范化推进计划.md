# AI 导演网页规范化推进计划

更新时间：2026-04-28

## 1. 推进目标

当前项目已经有可运行的 AI 导演成片工作台，但网页开发方式还需要规范化：不能再用一个大目标让 Agent 自己脑补页面，而要拆成“明确范围、明确输出、明确验收标准”的任务包。

今晚的目标不是大规模改页面，而是把后续网页开发变成稳定流程：

1. 学习 GitHub/开源生态里的网页开发规则、Agent 规则、视觉 QA、组件文档和审计工具。
2. 把这些能力转成当前项目可用的规范、提示词、任务拆分和验收清单。
3. 形成可以每半小时推进、每小时汇报、晚上交付的常规化任务机制。

## 2. 已更新的自动化节奏

已复用现有 `ai-7` 自动化，改成：

- 每 30 分钟推进一次。
- 默认半小时节点安静推进。
- 整点或接近整点向用户汇报。
- 今晚 21:00-22:00 重点交付完整规划。
- 不自动安装第三方插件、不运行新下载脚本；只学习、归纳、列候选，安装类动作以后单独确认。

## 3. GitHub/开源学习结论

### 3.1 Agent 规则库：awesome-cursorrules

来源：[PatrickJS/awesome-cursorrules](https://github.com/PatrickJS/awesome-cursorrules)

可借鉴机制：

- 给 Agent 写项目级规则，而不是每次临时解释。
- 规则要包含项目结构、技术栈、编码风格、禁止事项、测试要求。
- 规则要让 Agent 保持上下文一致，减少“乱造新结构”。
- 对当前项目最有用的是：开发前先读规格、改动前说明范围、完成后截图验收。

落地到本项目：

- 建立“AI 导演网页开发 Agent 规范”。
- 所有任务包都要求先读 `docs/AI导演工作台页面规格.md` 和本计划。
- 每个任务包只允许改和任务相关的区域。
- 每次交付必须写清楚：改了什么、怎么验收、还缺什么。

### 3.2 浏览器验收：Playwright

来源：[microsoft/playwright](https://github.com/microsoft/playwright)

可借鉴机制：

- 用真实浏览器打开页面，而不是只看代码。
- 覆盖桌面和手机视口。
- 对关键状态截图：空状态、生成中、已生成、失败/超时。
- 能做交互流程验证：输入想法、点击按钮、等待状态变化、检查页面文本和布局。

落地到本项目：

- 先不安装 Playwright 包，当前阶段用 in-app browser 做人工截图 QA。
- 后续如需自动化，再加 Playwright 脚本。
- 每个 UI 任务包至少检查 1440px 桌面和 390px 手机。

### 3.3 组件隔离和文档：Storybook

来源：[storybookjs/storybook](https://github.com/storybookjs/storybook)

可借鉴机制：

- 把 UI 组件或页面状态隔离出来看。
- 每个组件都有多个状态：默认、loading、empty、error、completed。
- UI 文档不是另写说明，而是和可运行示例绑定。

落地到本项目：

- 当前项目是原生 HTML/CSS/JS，不立刻引入 Storybook。
- 先用“状态样例清单”模拟 Storybook 思路。
- 每个模块都要列出必须展示的状态，避免只做 happy path。

### 3.4 可访问性检查：axe-core

来源：[dequelabs/axe-core](https://github.com/dequelabs/axe-core)

可借鉴机制：

- 自动检查常见可访问性问题。
- 关注标题层级、按钮名称、表单标签、颜色对比、ARIA 误用。
- 自动检查不能替代人工视觉 QA，但能发现大量基础问题。

落地到本项目：

- 短期人工检查：按钮必须有清晰文案，输入框必须有 label，状态不能只靠颜色表达。
- 后续可考虑加 axe-core 或 Storybook a11y addon。

### 3.5 Web 质量审计：Lighthouse

来源：[GoogleChrome/lighthouse](https://github.com/GoogleChrome/lighthouse)

可借鉴机制：

- 用性能、可访问性、最佳实践、SEO 等维度给网页做自动审计。
- 对当前内测工具来说，最重要的是性能和可访问性，不优先 SEO。

落地到本项目：

- 图片和视频素材不能无限加载，避免页面卡死。
- 大量 LibTV 结果要折叠展示，只加载代表素材。
- 后续可以把 Lighthouse 作为发布前检查，不作为当前首要开发任务。

## 4. 当前项目开发原则

### 4.1 先规格，后代码

每个任务开始前必须确认：

- 目标用户是谁。
- 本任务解决哪个具体痛点。
- 页面上新增或改变哪些区域。
- 成功后用户第一眼能看到什么。
- 哪些东西明确不做。

### 4.2 按成果拆，不按零碎操作拆

错误拆法：

- 调一下按钮颜色。
- 加一点间距。
- 改一个标题。

正确拆法：

- 完成“剧本可视化”模块：输入想法后展示标题、梗概、旁白、文字分镜；桌面和手机不溢出；能用自然语言调整但不触发 LibTV。

### 4.3 每个任务包必须可验收

每个任务必须写清楚：

- 必须包含什么。
- 哪些交互必须可用。
- 哪些状态必须覆盖。
- 桌面和手机怎么排布。
- 什么情况算失败。

### 4.4 截图反馈优先

网页任务不能只靠文字验收。每个关键任务完成后要看：

- 桌面截图。
- 手机截图。
- 空状态截图。
- 有结果状态截图。
- 长文本状态截图。

## 5. 任务包拆分

### 任务 0：网页开发 Agent 规范

目标：

建立后续所有 Agent 都要遵守的开发规则。

输出：

- 开发前阅读清单。
- 允许改动范围。
- 禁止事项。
- 完成汇报格式。
- 截图 QA 要求。

验收标准：

- 新 Agent 拿到规范后能知道先读哪些文档。
- 新 Agent 不会一上来大改结构。
- 新 Agent 知道每个任务必须截图验证。

建议文件：

- `docs/AI导演网页Agent执行规范.md`

### 任务 1：首屏和结果总览

目标：

输入想法后，右侧第一屏明确告诉用户“现在发生了什么、结果在哪里、下一步做什么”。

输出：

- 当前任务状态。
- 全流程步骤。
- 当前结果总览。
- 六个验收块：剧本、角色、场景、道具、分镜、成片。
- 每块一个“查看”入口。

验收标准：

- 输入想法后必须出现当前结果总览。
- 没有素材时也要显示草稿名称。
- 点击查看能跳到对应区域。
- 桌面端第一屏能同时看到输入区和结果总览。
- 手机端输入区后紧跟结果总览。

当前状态：

- 已有结果总览和查看入口。
- 下一步需要截图检查 1440px 和 390px。
- 需要弱化备用 provider 的错误状态，避免误导主链失败。

### 任务 2：剧本可视化

目标：

输入想法后，剧本不再是隐藏文本，而是可验收的导演文档。

输出：

- 标题。
- 故事梗概。
- 整片旁白。
- 文字分镜卡片。
- 剧本自然语言调整入口。

验收标准：

- GPT 返回后立刻显示标题、梗概、旁白和至少 3 条分镜。
- 每条分镜有镜头编号、标题、画面说明、旁白。
- “只调整步骤”能改剧本但不消耗 LibTV。
- 长文本不撑破卡片。

当前状态：

- 已有剧本区和文字分镜。
- 需要把“剧本自然语言调整”从全局导演指令中更明确地露出。

### 任务 3：角色/场景/道具主体图

目标：

让主体设定从文字卡片变成可见图像参考。

输出：

- 角色/演员卡。
- 场景卡。
- 道具卡。
- 本组自然语言调整。
- 单卡生成。
- 本组生成。
- 素材展示。
- 选用按钮。

验收标准：

- 每组至少一张可编辑卡。
- 保存修改可用。
- 单卡 LibTV 生成可用。
- LibTV 返回图后能显示。
- 多个图能选用一个。
- 选用图进入剪辑素材池和最终成片上下文。

当前状态：

- 已有卡片、生成、同步、选用。
- 需要补更清晰的“图像还未生成/已提交/已返回”状态。

### 任务 4：分镜到视频镜头

目标：

把文字分镜变成图片分镜和视频镜头，按顺序可验收。

输出：

- 分镜卡。
- 分镜时长。
- 镜头运动。
- 旁白。
- 单镜头生成。
- 按顺序生成分镜。
- 视频预览。

验收标准：

- 每条分镜都能看到时长、镜头、旁白。
- 每条分镜能单独生成。
- 返回视频后可播放。
- 时间线按分镜顺序显示状态。
- 未生成的视频镜头不能看起来像失败。

当前状态：

- 已有分镜卡和时间线。
- 需要补“按顺序生成并验收”的界面提示。

### 任务 5：剪辑面板

目标：

让用户像在剪辑软件里一样理解素材如何进入成片。

输出：

- 素材池。
- 预览窗口。
- 检查器。
- V1/A1 时间线。
- 剪辑自然语言指令。
- 用当前时间线成片。

验收标准：

- 素材池显示所有已返回素材。
- 已选用素材排在前面。
- 有最终视频时预览窗口播放最终视频。
- 没素材时有明确占位。
- 时间线不因内容变化乱跳。
- 手机端单列可用。

当前状态：

- 已有剪辑面板。
- 需要截图检查和视觉打磨。

### 任务 6：最终成片和同步

目标：

让 LibTV 成片过程可理解、可等待、可同步。

输出：

- 串成成片按钮。
- 成片等待状态。
- LibTV 画布链接。
- 手动同步按钮。
- 最终视频播放器。
- 错误提示。

验收标准：

- 提交后能看到等待状态。
- 返回视频后能播放。
- 超时但有画布链接时能继续同步。
- 错误信息面向用户，不暴露 key 或内部堆栈。

当前状态：

- 已有成片和同步。
- 需要优化超时/失败文案。

### 任务 7：截图 QA 和视觉修复

目标：

每次页面任务完成后，用固定截图矩阵验收，而不是靠感觉。

输出：

- 桌面 1440px 截图。
- 手机 390px 截图。
- 空状态。
- 生成中状态。
- 有剧本无素材状态。
- 有素材状态。
- 已成片状态。
- 长文本状态。

验收标准：

- 无横向滚动。
- 按钮文字不溢出。
- 卡片不重叠。
- 图片视频不撑坏布局。
- 第一屏能解释“现在发生了什么”。
- 备用 provider 错误不干扰主链判断。

## 6. Agent 提示词模板

### 6.1 通用开发任务模板

```text
你要在当前仓库里完成一个 AI 导演网页任务。

先阅读：
1. docs/AI导演工作台页面规格.md
2. docs/AI导演网页规范化推进计划.md

任务名称：
[填写任务包名称]

任务范围：
[只允许改哪些区域]

必须输出：
[列出页面上必须出现的内容]

必须交互：
[列出必须可点击/可保存/可跳转/可播放的动作]

验收标准：
1. 桌面 1440px 不溢出、不重叠。
2. 手机 390px 不溢出、不重叠。
3. 空状态、生成中、已生成状态都清楚。
4. 不暴露 API key。
5. 不新增无关功能。

完成后汇报：
1. 改了什么。
2. 怎么验证。
3. 截图里还有什么明显问题。
```

### 6.2 视觉 QA 模板

```text
请只做视觉 QA，不改业务逻辑。

检查页面：
http://127.0.0.1:8123/

视口：
1. 1440x900
2. 390x844

检查状态：
1. 无任务空状态
2. 有剧本无素材
3. 有素材
4. 已成片

检查项：
- 第一屏是否知道下一步做什么
- 文字是否溢出
- 按钮是否挤压
- 卡片是否重叠
- 视频/图片是否撑坏布局
- provider 状态是否误导

输出：
按严重程度列出问题，每条包含位置、原因、建议修复方式。
```

### 6.3 研究任务模板

```text
请研究一个网页开发或 AI 视频工具，不安装、不运行第三方代码。

研究对象：
[GitHub 仓库或产品]

输出：
1. 这个工具解决什么问题。
2. 适合当前 AI 导演网页的机制。
3. 不适合当前阶段的部分。
4. 可以转化成哪个任务包的验收标准。
5. 是否建议后续安装，安装前需要用户确认什么。
```

## 7. 今晚交付结构

今晚 21:00-22:00 的完整交付建议包含：

1. GitHub/开源工具学习摘要。
2. 当前网页差距分析。
3. 任务 0-7 的优先级排序。
4. 每个任务包的完成定义。
5. 视觉 QA 检查清单。
6. 下一天可执行的前三个小任务。

## 8. 优先级建议

第一优先级：

- 任务 0：Agent 执行规范。
- 任务 1：首屏和结果总览截图 QA。
- 任务 2：剧本可视化稳定性。

第二优先级：

- 任务 3：主体图生成和选用体验。
- 任务 4：分镜到视频镜头。

第三优先级：

- 任务 5：剪辑面板视觉打磨。
- 任务 6：最终成片同步文案。
- 任务 7：自动化截图 QA。

## 9. 暂不执行的事项

以下动作先不做，等用户确认：

- 安装 Playwright。
- 安装 Storybook。
- 安装 axe-core。
- 安装 Lighthouse。
- 引入新的前端框架。
- 重构为 React/Vite/Next。
- 大规模重写现有页面。

## 10. 下一步

下一步建议先写 `docs/AI导演网页Agent执行规范.md`，把任务执行规则固定下来。然后按任务 1 做一次桌面和手机截图 QA，列出首屏和结果总览的具体问题，再进入小步修复。
