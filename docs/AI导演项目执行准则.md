# AI 导演项目执行准则

## 1. 项目北极星

我们的目标不是先做一个“大而全”的 AI 影视平台，而是先做出一条真实可跑通的 Web 链路：

`一个想法 -> 结构化剧本 -> 旁白 -> 视频片段 -> 拼接导出 -> 成片预览`

所有决策都围绕这条链路服务。凡是不能明显提高这条主链成功率的工作，一律延后。

## 2. 我们的简化工作法

### 原则 1：先明确目标，再动手

每个任务都必须先写清楚“本次到底要验证什么”。

好目标：

- 输入一句 idea，返回稳定的 `story JSON`
- 接入 1 个真实视频 provider，能生成至少 1 个镜头
- 导出 1 条带旁白和字幕的 `mp4`

坏目标：

- 优化一下架构
- 顺手把前后端都重构了
- 先把系统设计完整

### 原则 2：先跑通最短链路，不提前抽象

默认优先顺序：

1. 先做能跑的版本
2. 再做稳定的版本
3. 最后再做优雅的版本

如果一个抽象没有立即减少重复、没有立即降低错误率，就先不要做。

### 原则 3：所有假设必须显式写出来

凡是需求、供应商、模型、用户路径存在歧义时，不要默认脑补，而是把当前假设写出来。

例如：

- 当前第一版是纯 Web，不做本地客户端
- 当前主视频链默认阿里百炼，Runway 作为备用
- 当前未配高质量 TTS 时，允许降级为系统语音

### 原则 4：每一步都要能被验证

每次推进都要有可验证产物，不能只停留在“分析”“构想”“差不多能行”。

可验证产物包括：

- 一个接口返回值
- 一条真实生成任务记录
- 一个中间文件
- 一个最终视频
- 一份明确的错误日志

### 原则 5：优先做手术刀式修改

改动默认应该尽量小、尽量集中、尽量可回退。

不因为一个局部问题，顺手改掉整套命名、目录结构、代码风格或 UI 体系。

### 原则 6：优先保主链，不被支线拖死

当前主链优先级固定：

1. 剧本生成
2. 视频生成
3. 旁白生成
4. 拼接导出
5. Web 展示

支付体系、复杂账号系统、多人协作、可视化时间线编辑器，全部属于后置项。

## 3. 当前产品边界

### 第一版一定要有

- 输入项目想法的页面
- 创建任务接口
- 任务状态查询
- 真实 provider 状态显示
- 至少 1 条可用的视频生成链路
- 旁白音频生成或降级方案
- 最终成片导出

### 第一版暂时不要做

- 复杂登录注册体系
- 用户付费系统
- 多人协作
- 时间线编辑器
- 本地客户端
- 过早追求电影级 UI 动效

## 4. 供应商策略

### 文本层

- 首选：`OpenRouter + gpt-5.4`
- 备选：`DashScope + qwen-plus`
- 兜底：本地模板生成

文本层的职责是：

- 想法扩写
- 剧本提案
- 分镜拆解
- 提示词整理
- 旁白文案

### 视频层

- 主链：`DashScope / 通义万相`
- 备用：`Runway API`
- 本地兜底：静态卡片 + Remotion 过场，仅用于演示流程，不视为真实视频能力

结论：文本模型不能替代视频模型。

### 剪辑层

- 模板与镜头组织：`Remotion`
- 拼接、混音、字幕、导出：`FFmpeg`

结论：不购买第三方自动剪辑 API，先把服务端链路跑通。

### 配音层

- 首选：`ElevenLabs`
- 兜底：`macOS say`

结论：没有 ElevenLabs 也能跑 demo，但成片质感会受影响。

## 5. 单次任务执行流程

每个任务都按下面 6 步推进：

1. 先写本次目标
2. 写清当前假设和依赖
3. 只做最短实现路径
4. 立刻做最小验证
5. 记录结果、问题和下一步
6. 再决定是否扩展

一个合格的任务完成状态，不是“代码写了”，而是“验证通过了”。

## 6. 我们的 Definition of Done

一个功能只有同时满足下面条件，才算完成：

- 能被真实触发
- 有可观察状态
- 失败时有明确报错
- 有中间产物或最终产物
- 用户能看懂当前进度

对于当前项目，典型完成标准是：

- 能创建任务
- 能看到当前阶段
- 能知道正在使用哪个 provider
- 失败时能看到原因
- 成功时能拿到可打开的结果文件

## 7. 当前阶段的正确推进顺序

### Phase 1：文字链路稳定

目标：

- 输入 idea
- 输出稳定 `story JSON`

### Phase 2：旁白链路稳定

目标：

- 生成整条片子的旁白音频

### Phase 3：真实视频链路打通

目标：

- 至少 1 个 provider 能产出真实镜头

### Phase 4：导出链路打通

目标：

- 将镜头、旁白、字幕合成为最终 `mp4`

### Phase 5：Web 体验打磨

目标：

- 状态页、结果页、错误反馈更清晰

## 8. 我们明确不做的坏习惯

- 不把“规划”当成“交付”
- 不把“写了很多代码”当成“功能完成”
- 不因为兴奋就扩展功能边界
- 不在没验证主链前做大重构
- 不把缺少外部 key 的问题伪装成代码问题

## 9. 协作约定

我接下来会按这套方式推进：

- 遇到歧义先说清假设
- 默认优先完成最小可验证版本
- 每一步都给你产物和结论
- 有真实阻塞就直接指出，不假装已经解决

这份准则的本质只有一句话：

`先把一条真实链路跑通，再谈平台化。`
